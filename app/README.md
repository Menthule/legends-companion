# Legends Companion desktop app (Tauri v2)

Dashboard + click-through overlays for the EverQuest Legends log companion.
React 18 + TypeScript + Vite frontend; Rust core wiring `eqlog-core`
(tail/parse/fights) and `eqlog-triggers` (trigger engine) into Tauri.

## Building

The frontend builds anywhere:

```sh
cd app
npm install
npm run build        # runs strict `tsc --noEmit` then `vite build`
```

The Rust side **must build on Windows** (WebView2, WinRT TTS, WASAPI audio).
`app/src-tauri` is intentionally *outside* the root cargo workspace (its
`Cargo.toml` has its own `[workspace]` table) so `cargo check` at the repo
root never touches it. On a Windows machine / windows-latest CI:

```sh
cd app
npm install
npm run tauri build        # NSIS installer (bundle.targets = ["nsis"])
# or, for checks only:
cd src-tauri && cargo check && cargo test && cargo clippy --all-targets
```

`cargo test` in `src-tauri` includes a fixture test that parses
`fixtures/sample_session.txt` through the real parser + fight tracker.

## Architecture

- **Windows** (tauri.conf.json): `main` dashboard, plus hidden `overlay-alerts`
  and `overlay-meter` (transparent, undecorated, always-on-top, skip-taskbar).
  All three load the same Vite app; overlays select their view via
  `?overlay=alerts|meter`.
- **Tail session** (`src-tauri/src/tailing.rs`): one worker thread runs
  Tailer â†’ Parser â†’ FightTracker.ingest + TriggerEngine.process. The
  `recv_timeout(250ms)` loop doubles as the tick that polls
  `TriggerEngine::due(now_ts)` (timers run on log-line time, extrapolated by
  wall clock between lines) and flushes throttled fight updates.
- **Audio** (`src-tauri/src/audio.rs`): dedicated thread; `tts` (WinRT) +
  `rodio` under `cfg(windows)`, log-only no-op elsewhere.
- **Events emitted to all windows**:
  - `log-line` `{ts, message, event}` â€” every parsed line (event = eqlog-core
    `Event`, serde externally-tagged)
  - `trigger-fired` `{trigger, action: {kind, text}}` â€” trigger is currently
    always `null` (ActionSink does not carry trigger identity)
  - `timer` `{name, kind: started|warning|expired, durationSecs?, warnAtSecs?}`
  - `fight-update` `{target, durationSecs, totalDamage, active, rows}` â€”
    throttled to ~2/sec, sourced from `FightTracker::active_fights()`
  - `overlay-lock-changed` `{label, clickThrough}` â€” mirrors the
    "unlock to arrange" toggle so overlays can show their drag handles
- **Commands**: `get_config`, `set_config`, `get_triggers`, `save_triggers`,
  `import_gina(path) -> {imported, warnings}`, `start_tailing`, `stop_tailing`,
  `is_tailing`, `overlay_show/hide(label)`,
  `overlay_set_click_through(label, ignore)`.
- **Storage**: `config.json` and (by default) `triggers.json` in the Tauri app
  config dir. Trigger packs are saved in the `TriggerPack` JSON shape from
  `eqlog-triggers::model`; bare `[Trigger, ...]` arrays are accepted on read.

## Icons

`src-tauri/icons/` is generated by `gen_icons.py` (stdlib-only Python,
committed output). Tauri requires RGBA PNGs + `icon.ico`; regenerate with
`python3 gen_icons.py` from that directory.

## Mock mode (browser-only visual verification)

The frontend runs standalone in any browser â€” no Tauri backend needed. Mock
mode activates automatically when `window.__TAURI_INTERNALS__` is absent, or
explicitly with `?mock=1`. A scripted driver (`src/mock.ts`) replays a
realistic session derived from `fixtures/sample_session.txt`: a live fight
against Baron Telyx V`Zher (4 combatants, updates every 500ms, rank 3/4 trade
places to exercise re-sorting), three countdown timers (one inside its warn
threshold; timers expire and restart to exercise the pulse/fade), a ~40-line
mixed live feed, alert pills, and the starter trigger pack.

Serve it with:

```sh
cd app && npm run build && npx vite preview --host 0.0.0.0 --port 4173
```

URL parameters:

- `?mock=1` â€” force mock mode (implied outside Tauri)
- `&tab=live|meters|triggers|settings` â€” initial page
- `&theme=dark|light` â€” force a theme for the session (dark is default;
  the Settings toggle persists to localStorage)
- `?overlay=alerts` / `?overlay=meter` â€” overlay windows
  (`?window=overlay-alerts|overlay-meter` also accepted)
- `&unlocked=1` â€” overlays start in the unlock-to-arrange state; in mock mode
  a small lock/unlock button is also shown bottom-right
